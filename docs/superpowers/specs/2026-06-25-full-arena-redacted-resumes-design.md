# Full Arena-Branded Redacted Résumés — Design

**Date:** 2026-06-25
**Status:** Awaiting user review

## Problem

The redacted résumé PDFs attached to Arena 4x4 campaigns are thin and unprofessional
(see current output: a "Candidate C — Redacted Resume" with a role line and ~6
bullets). The user's target is a full, Arena-branded professional résumé like the
`Trevor_Myers_Arena.pdf` / `Matthew_Collins_Arena.pdf` examples: logo, professional
summary, two-column core competencies, full professional experience with **real
employers and dates**, certifications, and education.

Root cause is a **data problem, not just formatting**: the AICB candidate *cards*
carry only `label` + `role` + a few summary `bullets`. The full résumé
(`resume_text`, work history with employers/dates) lives in the candidate pool / ATS
for pool-sourced candidates but is **dropped** when the card is built
([flowdrip_app.py:31959-31964](../../flowdrip_app.py)). AI-autogenerated
("Candidate A/B/C") candidates have no résumé on file at all. The current PDF can
only render what's on the card, so it comes out thin.

## Goal

Generate a full, Arena-branded redacted résumé per candidate that contains real
employers, titles, dates, certifications, and education — with **only the
candidate's name and contact info removed** — matching the example PDFs.

## Decisions (locked with user, 2026-06-25)

- **Source = Both, automatically.** Real ATS/pool résumé when the candidate carries
  `resume_text`; AI-expand the card's role + blurb when it doesn't (autogen).
- **Redaction = name + contact only.** Keep real employers, titles, dates, certs,
  education, and city/state. Strip name, email, phone, LinkedIn/URLs, street address.
- **No AI tells.** No "representative profile" footnote, no "contact information has
  been removed" auto-note, no marker distinguishing real vs AI-expanded. The résumés
  render clean and identical regardless of source. The redacted identity (the
  "Candidate A" label shown where the name would be, with no contact block) is the
  only signal — standard recruiter blind-submittal practice.
- One extra AI call per candidate (to structure the résumé) is acceptable.

## Non-Goals

- Pixel-perfect reproduction of the example PDFs (reportlab approximates the layout;
  structure + branding match closely).
- No change to the email blurb generation or to the Candidate Résumés picker
  (shipped 2026-06-24) — filenames stay `Resume_<label>_Redacted.pdf`.

## Architecture

One normalized schema, two source adapters that produce it, one renderer that
consumes it. This keeps real and AI-expanded résumés visually identical and makes
the renderer a pure, testable function.

### 1. Normalized résumé schema (`ResumeDoc`)

```python
ResumeDoc = {
  "label": "Candidate A",        # shown where the name goes (redacted identity)
  "headline": "Manufacturing Manager",
  "location": "Windsor, CO",     # city/state kept
  "summary": "...",              # professional summary paragraph
  "competencies": ["...", ...],  # CORE COMPETENCIES (rendered two-column)
  "experience": [
     {"company": "Helena Agri-Enterprises", "title": "Plant Manager",
      "dates": "2018 – Present", "location": "Windsor, CO",
      "bullets": ["...", "..."]},
  ],
  "earlier_experience": "",      # optional condensed one-paragraph line
  "certifications": ["...", ...],
  "education": [{"school": "...", "detail": "...", "location": "..."}],
}
```

A `_normalize_resume_doc(d)` helper coerces missing keys to safe defaults so the
renderer never KeyErrors on a partial AI response.

### 2. Source adapters (both in `flowdrip_app.py`, near existing redaction code ~L38622)

- `_resume_doc_from_text(client, resume_text, known_name, label, headline, location)`
  — **real path.** Claude structures `resume_text` into `ResumeDoc` JSON. Uses the
  existing `_wrap_untrusted("resume_text", resume_text, ...)` injection guard. The
  prompt is told to preserve employers/titles/dates verbatim and to OMIT name/contact.
- `_resume_doc_from_card(client, card)` — **representative path.** Claude expands the
  card's `role` + `bullets` into a full plausible `ResumeDoc` (realistic employers,
  dates, certs, education) consistent with the blurb. Header label = card `label`.

Dispatch in `_build_candidate_resume_doc(client, card)`:
```
text = (card.get("resume_text") or "").strip()
return _resume_doc_from_text(...) if text else _resume_doc_from_card(...)
```

### 3. Carry résumé data onto pool cards (the leak fix)

In the pool→card builder ([flowdrip_app.py:31959](../../flowdrip_app.py)) add
`resume_text`, real `name`, and `location` from the pool candidate to the card dict.
No schema change elsewhere — autogen cards simply won't have `resume_text`.

### 4. Deterministic redaction pass — `_redact_resume_doc(doc, known_name)`

Belt-and-suspenders after AI structuring (never trust the model alone):
- Remove `known_name` tokens (first/last, case-insensitive) from every text field.
- Strip emails (`\S+@\S+`), phone numbers, `linkedin.com/...` and any `http(s)://`
  / `www.` URLs, and street-address lines.
- **Keep** city/state, employers, titles, dates, certs, education.
- Applied recursively to `summary`, every `experience[].bullets`, `earlier_experience`,
  `education[].detail`.

### 5. Renderer `build_resume_pdf(output_path, doc)` in `funnel_forge/arena_pdfs.py`

Pure function (no AI). Matches the examples using existing primitives
(`section_header()`, `bullet_item()`, `alt_table()`) plus:
- `Arena10Logo.png` (already in `assets/`) top-right + name header centered
  (= `doc["label"]`), headline + `location` subtitle in Arena blue.
- `PROFESSIONAL SUMMARY` — paragraph.
- `CORE COMPETENCIES` — two-column `Table` of `competencies`.
- `PROFESSIONAL EXPERIENCE` — per entry: bold company (left) + right-aligned dates
  on one row, italic-blue title + right-aligned location on the next, then bullets.
- `EARLIER EXPERIENCE` — single paragraph, only if present.
- `CERTIFICATIONS` — bullets.
- `EDUCATION` — per entry: bold school (left) + right-aligned location, italic-blue
  detail beneath.

No redaction note, no footnote.

### 6. Wire-in — `_aicb_build_redacted_resumes` ([flowdrip_app.py:30776](../../flowdrip_app.py))

Replace the thin `_aicb_card_to_resume_text` → `_save_redacted_pdf` path with:
```
doc = _build_candidate_resume_doc(client, card)
doc = _redact_resume_doc(doc, known_name=card.get("name", ""))
build_resume_pdf(_user_pdf_dir() / f"Resume_{slug}_Redacted.pdf", doc)
```
Runs in the existing AICB generation worker (already `_run_as_user`-dispatched, so
`_user_pdf_dir()` resolves per-user). Filenames unchanged → picker still finds them.

`_aicb_card_to_resume_text` loses its only caller and is removed. **`_save_redacted_pdf`
is KEPT** — it has a second caller in the CPC (candidate placement) flow
([flowdrip_app.py:38858](../../flowdrip_app.py)) which still relies on the thin
text→PDF path. See "Out of scope" below.

## Out of Scope (follow-up)

The CPC flow ([flowdrip_app.py:38851-38869](../../flowdrip_app.py)) generates the same
thin redacted-résumé PDF from an already-redacted `redacted_resume` text and
auto-attaches it to emails 1/3/5. It has the identical quality problem, but it is a
different campaign type with different (auto-attach) behavior and its own data path.
`build_resume_pdf` + the `ResumeDoc` adapters are written reusably so upgrading CPC
later is small. Not changed in this spec to avoid altering CPC's send behavior
without explicit sign-off.

## Data Flow

```
pool candidate ─► card now carries resume_text/name/location ─┐
autogen card (role+bullets, no resume_text) ─────────────────┤
                                                              ▼
                        _build_candidate_resume_doc(client, card)
                     real path (resume_text) │ representative path (no text)
                                              ▼
                                 ResumeDoc (AI JSON)
                                              ▼
                       _redact_resume_doc(doc, known_name)   ← deterministic strip
                                              ▼
                build_resume_pdf → Resume_<label>_Redacted.pdf (per-user PDFs/)
                                              ▼
                 Candidate Résumés picker (View / Attach) — unchanged
```

## Error / Edge Handling

- **AI returns bad/partial JSON** → `_normalize_resume_doc` fills defaults; renderer
  still produces a valid (if sparser) PDF. One bad candidate never aborts the batch
  (existing per-card try/except in `_aicb_build_redacted_resumes` stays).
- **Empty résumé text on a pool card** → falls through to the representative path.
- **Prompt injection in `resume_text`** → wrapped via existing `_wrap_untrusted`.
- **Name leak** → deterministic `_redact_resume_doc` removes known-name tokens even
  if the model echoes them.
- **Missing logo asset** → existing `arena_pdfs` text wordmark fallback.

## Testing

Pure-function unit tests (no UI, no live AI):
- `_normalize_resume_doc`: partial dict → all keys present with safe types.
- `_redact_resume_doc`: strips name tokens, emails, phones, LinkedIn/URLs; KEEPS
  employers, dates, city/state, certs.
- `_build_candidate_resume_doc` dispatch: card with `resume_text` → real path; card
  without → representative path (assert via a monkeypatched/fake client).
- `build_resume_pdf`: given a full `ResumeDoc`, writes a non-empty PDF file that
  contains expected section markers (smoke check on byte output / reportlab story).

Manual: generate a 4x4 with (a) a pool candidate and (b) an autogen candidate; open
both résumés from the picker; confirm full sections, real employers/dates on the pool
one, no name/contact on either, clean look with no AI/redaction notes.
